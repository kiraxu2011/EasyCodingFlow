# 多框架融合ecf skills开发思路

## 1、分层架构：

### 0层：

编排层 -- 规范契约层 -- 执行层 -- 知识沉淀层

### 1层：

编排层：意图识别 -- 流程编排 -- 团队组建

规范契约层（OpenSpec）：需求分析 -- 明确规格 -- 变更管理

执行层（Superpowers）：设计 -- 任务拆解 -- 开发 -- Review -- 测试

知识沉淀（Compound Engineering Plugin）：总结复盘 -- 知识库管理

## 2、编排层

1）需要能根据用户输入，进行意图识别，判断任务复杂度，然后走不同的流程。

软件开发过程中常见的任务如下：

新需求开发

增量/迭代开发

Bug修复

代码审查（Code Review）

代码重构

文档更新

DT用例补齐

2）使用EasyCodingFlow能力组建团队

## 3、易用性优化

1）团队构建后，支持自定义指定不同任务使用不同的模型，或者可以在全局的配置文件中去指定。（EasyCodingFlow中都是Anthropic的模型）

2）将全流程细化成具体的子阶段，支持通过配置文件指定子阶段使用的skills，提供默认配置。

## 参考知识

### Superpowers ：**强纪律与反冲动机制**

https://github.com/obra/superpowers

Superpowers 的核心设计理念是建立最强的流程约束与反冲动机制，旨在将 AI 训练成“守纪律的工程师”。其设计目标不是单纯地增加工作步骤，而是强制开发过程的正规化。该项目明确要求创意工作必须先进行头脑风暴，且未经批准不得实施，强调 TDD（测试驱动开发）、YAGNI（你不会需要它）和 DRY（不要重复自己）等工程原则。

- **流程控制与技能化**废弃了早期的薄命令包装，转而强制使用 Skills 进行流程控制。核心技能包括 superpowers:brainstorming、superpowers:writing-plans 和 superpowers:executing-plans。
- **严格的实施计划规范**要求生成的实施计划必须能让零上下文、品味较差且测试设计能力弱的工程师按图施工。计划被视为可执行的施工图，而非讨论稿。
- **职责拆分与多 Agent 机制**每个任务使用新的 Subagent，实现后必须经过 Spec Review 和 Code Quality Review 两层审查。系统明确定义了实现者的状态机（DONE、DONE_WITH_CONCERNS、NEEDS_CONTEXT、BLOCKED），并鼓励根据任务复杂度选择能完成任务的最弱模型。
- **极端的 TDD 约束**明确规定“没有失败的测试之前不准写生产代码”，并将 RED/GREEN 验证作为强制步骤。

Superpowers 的 Prompt 设计风格偏向操作规范，充满了动作约束，旨在限制模型擅自行动，而非鼓励其自由发挥。

### OpenSpec：**规格化与结构化变更管理**

https://github.com/Fission-AI/OpenSpec

OpenSpec 的设计哲学追求流畅、迭代、简单且优先适应已有代码库。它不是教条化的瀑布流大流程，而是专注于在已有代码库中进行持续的变更管理。其核心关注点在于“变更如何被表示、推进、验证、合并、归档”，而非单纯的模型行为控制。

- **核心命令流**提供了一系列明确的命令来管理变更生命周期：/opsx:propose 直接创建变更套件；/opsx:explore 仅进行探索对话；/opsx:new 创建空骨架；/opsx:continue 按依赖图生成下一个工件；/opsx:ff 按依赖顺序一口气生成全部工件；/opsx:apply 落地执行规格；/opsx:archive 进行归档。
- **依赖图驱动**其流程并非线性固定阶段，而是基于 Artifact DAG（有向无环图）推进，这使得变更管理更加灵活且符合工程实际。
- **可配置的 Prompt 生成**Prompt 并非手工写死，而是通过 schema、context、rules 和 DAG 驱动的可配置生成机制。支持在 config.yaml 中配置注入规则，并明确区分不同宿主（如 Claude、Cursor）的命令语法。

OpenSpec 更像一个规格工作流层，适合在 Brownfield（既有代码库）场景下进行长期、结构化的变更追踪。

### **Compound Engineering Plugin：知识复利与闭环沉淀**

https://github.com/EveryInc/compound-engineering-plugin

Compound Engineering Plugin 强调“复利沉淀”，认为每一单位工程工作都应让后续工作更容易。其核心逻辑是将 80% 的精力投入 planning 和 review，仅 20% 用于 execution，并通过闭环机制将解决过的方案转化为长期资产。

- **需求与计划分离**Requirements 作为单一真理来源，Plan 负责决策与结构但不写代码。计划按复杂度分为 lightweight、standard 和 deep 三级，包含实现单元、系统影响、风险及文档备注。
- **工程化审查流水线**Review 环节是一个分阶段、多人格、置信度门控的流水线。系统会自动去重、autofix 并生成报告，比单纯的 Code Review 更加工程化和结果导向。
- **解决方案沉淀**独特的 ce:compound 功能将刚解决的问题沉淀到 docs/solutions/，并自动维护解决方案集合。通过 Keep / Update / Consolidate / Replace / Delete 等操作更新知识库，利用 git history 作为归档，确保知识的持续复用。

Compound 适合任务重复发生、需要知识沉淀且希望文档资产增值的团队，其核心优势在于完成后的资产积累而非单纯的执行。
